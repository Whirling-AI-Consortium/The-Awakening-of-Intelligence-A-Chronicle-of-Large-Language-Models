Prompt Engineering：对话的新语法

从微调到对话的范式转变​

在GPT-3发布的那一刻，一个新的工程问题应运而生。如果模型能够通过少量示例学习，那么关键就变成了：如何最有效地编写这些示例和指令？这个问题在学术界和工业界引发了一场新的探索热潮。

传统的NLP工作流中，工程师们的主要任务是微调（Fine-tuning）。给定一个预训练模型，收集标注数据，进行几个epoch的梯度下降，输出一个针对特定任务的模型。这个过程虽然有效，但有几个问题：需要足够的标注数据、微调后的模型只能处理单一任务、重新训练需要大量计算资源。

但GPT-3改变了这个游戏规则。现在，工程师的任务不再是调整参数，而是调整提示（Prompt）。这是一个本质的不同。参数是低维的数值空间，而Prompt是高维的自然语言空间。如何在这个新空间中进行"工程"成了一个新的学科。

这个转变在2021年和2022年变得越来越明显。大量的论文开始研究如何构建更有效的Prompt。OpenAI发布了关于Prompt设计的最佳实践指南。公司和研究机构开始雇佣"Prompt工程师"。整个社区开始积累关于如何与大模型交互的隐性知识。

从离散到连续的探索​

Prompt的形式可以沿着一个光谱分布。最直接的是自然语言的离散Prompt，比如"翻译这句话到法语"或"总结以下文章"。但研究者们很快意识到，也许可以做得更好。

一个重要的研究方向是参数高效的微调（Parameter-Efficient Fine-Tuning）。其中最著名的两个方向是Prefix Tuning和P-tuning。

Prefix Tuning的思想来自Stanford的Li和Liang。与其微调所有的模型参数，不如在输入序列的前面"冻结"一些特殊的token，只训练这些token对应的嵌入向量。这些特殊token构成了"前缀"，可以被看作是一个可学习的Prompt。数学上，如果原始的输入是 x1,x2,...,xnx_1, x_2, ..., x_nx1​,x2​,...,xn​，那么Prefix Tuning就是学习一个前缀 p1,p2,...,pmp_1, p_2, ..., p_mp1​,p2​,...,pm​，使得模型的注意力机制会优先考虑这个前缀中的信息。

代码上，这个想法可以这样实现：

class PrefixTuning(nn.Module):
    def __init__(self, model_dim, prefix_len, num_layers):
        super().__init__()
        # 只训练这些前缀向量，而不是整个模型
        self.prefix_embeddings = nn.Parameter(
            torch.randn(num_layers, prefix_len, model_dim)
        )
        
    def forward(self, hidden_states):
        # hidden_states: (batch_size, seq_len, model_dim)
        # 在序列开头附加前缀
        batch_size = hidden_states.size(0)
        prefix = self.prefix_embeddings.unsqueeze(0).expand(
            batch_size, -1, -1, -1
        )
        return torch.cat([prefix, hidden_states], dim=1)

这个方法的妙处在于，前缀本身不是自然语言，而是向量空间中的连续值。它比自然语言的离散Prompt更灵活，因为梯度可以流经这些连续向量。但它仍然保留了参数高效的优点——不需要训练数十亿的参数。

P-tuning走了类似但稍有不同的路。它不是在输入开头添加前缀，而是在输入的各处插入可学习的"虚拟token"。这些虚拟token没有对应的真实词语，但可以被优化以改进任务性能。

这些方法的出现反映了一个更深层的认识：Prompt不必是人类可读的自然语言。机器可以学到自己的"语言"，一种在向量空间中表达的指令。这打开了一个新的可能性——通过优化算法自动生成最优的Prompt。

指令遵循的艺术​

尽管Prefix Tuning和P-tuning展现了可学习Prompt的力量，但对于大多数实践者来说，手工编写自然语言Prompt仍然是主流方式。而这带来了一个新的挑战：如何让模型更好地理解和遵循指令？

在GPT-3的早期使用中，人们意外地发现，不同的表述方式会产生显著不同的结果。这不是因为句子的语义改变了（从信息论的角度），而仅仅是措辞的不同。例如：

  • "翻译这句话到法语：Hello" 的效果不如 "请用法语翻译以下英文句子。英文：Hello 法语："
  • "这个句子的情感是什么" 的效果不如 "以下句子表达了什么情感？可能的情感有：正面、负面、中立。句子：..."

这个现象引发了一个问题：为什么措辞的细微变化会影响模型的性能？一个可能的解释是，大模型在预训练过程中看到了大量不同风格的指令和问题。当Prompt的表述方式更接近预训练数据中常见的模式时，模型就能更好地"理解"。

这引发了Prompt工程中的一个关键原则：一致性和明确性。好的Prompt应该清晰地传达任务意图，提供充分的上下文，使用模型在预训练数据中经常遇到的表述方式。

另一个重要的发现是分步骤（Step-by-step）的力量。让模型在生成最终答案之前明确地描述其思考过程，往往会提高准确性。这与我们下一节将重点讨论的Chain-of-Thought密切相关，但在Prompt Engineering的背景下，它更多地是一种实用技巧而非理论框架。

论文《The Power of Scale for Parameter-Efficient Prompt Tuning》来自Google Brain的Brian Lester等人，系统性地研究了Prompt Tuning（一种连续Prompt的方法）在大模型上的有效性。他们的关键发现是，当模型规模达到某个阈值（大约100B参数）时，Prompt Tuning的性能能够与全量微调相当，同时只需要训练模型的0.01%的参数。这个结果有深刻的含义：对于足够大的模型，学习如何表达问题，几乎和学习模型本身一样重要。

示例的选择与排序​

在Few-Shot设置中，选择哪些示例会显著影响结果。这不是一个统计学的问题（不需要大量样本来估算参数），而是一个组合和启发式的问题。

早期的一个发现是，示例的顺序重要。模型对最后看到的示例更敏感。这被称为近期性偏见（Recency Bias）。如果你在Prompt的末尾放置一个特别相关的示例，模型在生成答案时就会更多地受到它的影响。

另一个发现是，示例的多样性重要。如果所有示例都来自同一个特定的"角度"或特征，模型可能会过度拟合到这些特征。相反，提供涵盖问题空间中不同区域的示例，往往会带来更好的泛化性能。

这引发了一些研究工作，致力于自动选择最优的示例集合。一个简单的策略是相似度匹配：给定一个新的输入，从候选示例集合中选择与这个输入最相似的K个示例。相似度可以通过各种方式测量，比如句子的词重叠或更复杂的语义相似度。

还有一个有趣的方向是对比性学习。不是提供最相似的示例，而是提供既相似又有所不同的示例对。这样的"对比"可以帮助模型更好地理解任务的边界和细节。

温度、采样与多样性控制​

Prompt的不仅仅是内容，也包括生成策略。当模型生成文本时，每个时刻它都需要从可能的下一个词中选择。通常，模型会计算每个词的概率分布，然后选择概率最高的词（贪心策略）或根据概率分布进行采样。

这个过程可以通过一个称为"温度"（Temperature）的参数来调节。温度是一个在0到1之间的值（虽然技术上可以超过1）。低温（接近0）使分布变得更"尖锐"，高概率的词被选中的可能性大幅增加。高温（接近1或更高）使分布变得更"平坦"，即使是较低概率的词也有机会被选中。

数学上，如果原始的logit分布是 z=[z1,z2,...,zV]\mathbf{z} = [z_1, z_2, ..., z_V]z=[z1​,z2​,...,zV​]，那么经过温度调节后的概率是：

pi=exp⁡(zi/T)∑jexp⁡(zj/T)p_i = \frac{\exp(z_i / T)}{\sum_j \exp(z_j / T)}pi​=∑j​exp(zj​/T)exp(zi​/T)​

其中 TTT 是温度。

在实际应用中，这意味着什么？对于创意任务（比如故事生成、诗歌创作），使用较高的温度能产生更有趣、更多样的结果。对于需要准确性的任务（比如事实问答、代码生成），使用较低的温度能产生更稳定、更一致的结果。

相关的还有Top-K采样和Nucleus采样（也叫Top-p采样）。这些方法不是调节整个分布，而是只考虑最有可能的K个词（Top-K）或累积概率超过某个阈值p的词（Top-p）。这样可以避免模型选择过于不太可能的词，同时保留多样性。

角色扮演与上下文设定​

一个有趣的Prompt工程技巧是角色扮演。通过告诉模型它扮演某个角色，可以显著改变其输出风格和内容。例如：

"你是一位资深的Python开发者。请解释这段代码的作用"

对比：

"请解释这段代码的作用"

第一个版本会倾向于使用更专业的术语、更深入的分析。模型似乎"理解"了它应该采用专家的视角。

这个现象有几种可能的解释。一是，模型在预训练数据中看过大量这样的角色扮演，因此能够从这个框架推断出适当的风格。二是，角色描述激活了模型中与这个角色相关的知识簇。第三，这可能是一种更细致的上下文设定形式。

相关的技术包括提供虚拟的对话历史、设定场景（"假设你在一个中世纪的魔法学院"）、或指定特定的格式和结构。这些都属于更广泛的"上下文工程"范畴。

链式思维与分解​

虽然Chain-of-Thought会在下一节得到更详细的讨论，但在Prompt Engineering的背景下，它首先体现为一种实用的Prompt设计模式。

最初的观察来自于一个简单的实验。当要求模型解决一个复杂的数学问题时：

"问：一个班级有36个学生。如果每个学生需要3个苹果，那么总共需要多少个苹果？答："

模型的答案往往是错误的。但如果改成：

"问：一个班级有36个学生。如果每个学生需要3个苹果，那么总共需要多少个苹果？ 让我们分步骤思考： 第一步：确定学生人数... 答："

模型的准确率显著提高。看起来，强制模型"显示工作"可以改进其推理能力。

这个观察激发了一个Prompt工程的最佳实践：分解复杂问题。对于涉及多个步骤或需要推理的问题，明确要求模型分步骤地思考往往能提高正确率。

负样本与限制条件​

好的Prompt不仅要说明模型应该做什么，还要说明模型不应该做什么。这包括：

禁止事项的明确说明。例如，在信息检索任务中明确说明"不要编造信息"。在翻译任务中说明"保留专有名词"。

错误示例的包含。除了提供正确的示例，提供一些错误的示例以及为什么它们是错误的，可以帮助模型理解任务的边界。虽然这需要更多的token，但在复杂任务中往往是值得的。

输出格式的严格规范。明确指定输出应该采用什么格式（JSON、Markdown、纯文本等）可以减少模型生成意外格式的概率。

一个综合的例子可能看起来是这样的：

你是一个信息提取助手。你的任务是从给定的文本中提取特定的信息。

规则：
1. 只提取文本中明确提到的信息
2. 如果信息不在文本中，回答"未在文本中找到"
3. 保持原始的表述，不做推断

错误示例：
文本："John在2023年完成了他的学位"
问："John何时开始他的学位？"
错误答案："大约四年前，假设这是一个标准的学位课程"
正确答案："未在文本中找到"

现在请处理以下任务：
文本：[输入文本]
问：[问题]
答：

迭代与反馈循环​

Prompt Engineering本质上是一个迭代过程。很少有人能一次就写出完美的Prompt。相反，工程师会尝试一个Prompt，观察结果，识别失败的模式，然后改进Prompt。

这个过程可以通过建立一个评估集来加速。选择一些代表性的测试用例，对每个修改后的Prompt进行评估，观察指标的变化。这样可以系统性地改进，而不是盲目地调整。

一些工具和框架已经被开发出来来支持这个过程。LangChain是一个流行的框架，提供了模块化的Prompt构建、链式调用、记忆管理等功能。Prompt Playground是一个交互式工具，允许用户快速尝试不同的Prompt，实时看到结果。

这些工具反映了一个现实：Prompt Engineering不再是一个技巧或技术细节，而是一个成熟的工程学科，有自己的工具、最佳实践和职业角色。

从工程到科学​

虽然Prompt Engineering最初看起来像是一个"手工艺"活——通过试错和直觉来改进Prompt——但它正在逐渐演变为一个更科学的领域。

研究者们开始进行系统的研究，尝试理解Prompt的各个方面如何影响输出。例如，论文《Prompting as Inverse Language Modeling》提出，Prompt可以被看作是Inverse Language Modeling问题——而不是预测下一个词，我们需要预测哪个Prompt会导致所需的输出。

其他研究关注于自动化的Prompt优化。贝叶斯优化、强化学习等技术被应用来自动搜索最优的Prompt。虽然这些方法有时会产生人类难以理解的Prompt，但它们揭示了Prompt空间中可能的最优点。

这个转变表明，Prompt Engineering不仅仅是一个临时的解决方案（直到我们有更好的方法），而是深度学习中一个基本的方面。当模型变得更大、更通用时，如何有效地与它们交互的问题就变得更加重要。

展望与开放问题​

Prompt Engineering仍然是一个快速演变的领域。有许多开放的问题：

为什么某些Prompt会更有效？到底是什么特征使一个Prompt成为好的Prompt？

是否存在一些通用的Prompt模式可以跨越不同的模型和任务？

能否自动学习最优的Prompt，而不需要人工干预？

当模型本身不断变化（新版本发布）时，Prompt的鲁棒性如何保证？

这些问题推动了后续的研究。但无论答案如何，一个事实已经清晰：在大型语言模型的时代，学会与模型对话——用Prompt这种新的"语言"——已经成为了一个关键的技能。

从某种意义上说，Prompt Engineering代表了人与AI之间的新的交互范式。微调时代，人们通过标注数据来"教"模型。Prompt工程时代，人们通过语言来"对话"。这个转变既是技术上的进步，也是哲学上的。它暗示，也许AI的未来不是隐藏在黑盒中的参数，而是通过对话、通过自然语言来塑造的。
